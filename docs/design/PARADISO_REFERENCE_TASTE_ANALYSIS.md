## 4. 레퍼런스 취향 DNA

사용자의 12개 레퍼런스를 통해 추출되는 취향의 본질적 특성은 다음 10가지로 요약된다.

1. **타이포그래피 주도형**: 레이아웃이 이미지나 일러스트레이션이 아니라 활자 위계와 크기 대비에서 시작된다. 텍스트 없이는 디자인이 성립하지 않는 구조.
2. **에디토리얼 그리드**: 단순 카드 나열보다 컬럼·행·인덱스 체계를 기반으로 편집한다. 신문이나 잡지 레이아웃의 DNA가 있다.
3. **웹사이트 우선, 앱 우선이 아님**: 데스크톱 웹의 넓은 화면을 기준으로 설계하며, 모바일은 축소판이다. 큰 터치 타깃, 바텀 내비게이션, 챗 버블은 이 취향과 충돌한다.
4. **카드 최소화**: 정보 단위를 네모 박스에 과도하게 격리하지 않는다. 행(row)과 디바이더로 구분하는 것이 기본값이다.
5. **인덱스형 행 구조**: 목록 아이템은 수평으로 스캔할 수 있는 행으로 제시된다. 좌·중·우 정보 분류가 자연스럽게 읽힌다.
6. **얇은 디바이더**: 카드 그림자, 두꺼운 테두리 대신 1px 내외의 얇은 선으로 섹션과 행을 나눈다.
7. **웜 뉴트럴 표면**: 순백색보다 약간 따뜻한 오프화이트나 웜 그레이가 기본 배경이다. 차갑지 않지만 감성적이지도 않다.
8. **미묘하지만 독특한 비례**: 눈에 띄지 않지만 표준적이지 않은 여백·크기 비례를 사용한다. 너무 작지도 너무 크지도 않은, 의도적 스케일.
9. **디자인 스튜디오식 절제**: 불필요한 장식을 추가하지 않는다. 디테일을 최소화하되 빠뜨리지 않는다. 과시적 UI보다 정확한 UI.
10. **공공성 신뢰 감각**: 정부 포털처럼 딱딱하지 않으면서도 시빅-테크 문맥에서 신뢰를 주는 톤. 과잉 친절하지 않고, 과잉 관료적이지 않다.

## 5. 공통 취향 요약

레퍼런스 전반에서 관찰되는 취향은 다음과 같다.

- **타이포그래피 주도형 구성**: 시각적 중심을 이미지가 아니라 텍스트 위계와 활자 밀도에 둔다.
- **에디토리얼 그리드 기반**: 카드 나열형 UI보다 열/행, 인덱스, 리스트 구조를 선호한다.
- **절제된 개성**: 장식이 많지 않지만 간격, 라인, 비례, 타이포 디테일로 캐릭터를 만든다.
- **따뜻하지만 귀엽지 않은 표면감**: 차갑게 기술적이거나 과도하게 감성적인 톤 대신, 온도를 약간 올린 중성 계열을 선호한다.
- **스튜디오식 긴장감**: 명확한 정보 구조 위에 정교한 편집 감각을 얹는 방식.
- **웹사이트 우선 사고**: 모바일 앱 패턴(큰 카드, 둥근 컴포넌트, 챗 UI)보다 데스크톱 웹 편집 체계 중심.

## 6. 레이아웃 취향 분석

### 6.1 카드 최소화와 인덱스화
사용자 취향은 "많은 카드"보다 "적은 블록 + 정교한 행 구조"에 가깝다. 즉, 정보 단위를 박스에 과도하게 격리하지 않고, 얇은 구분선과 여백으로 구획한다.

### 6.2 에디토리얼 그리드
단일 컬럼 스크롤보다 다음 구조를 선호한다.
- 좌측: 섹션 인덱스/내비게이션
- 중앙: 본문 정보(타이틀, 설명, 상태)
- 우측: 메타 정보(업데이트, 카테고리, 참조)

이는 문서적 신뢰감과 탐색 효율을 동시에 제공한다.

### 6.3 리듬 있는 밀도
콘텐츠 밀도가 낮아 텅 빈 랜딩페이지도, 반대로 높은 밀도의 법률 DB 느낌도 피해야 한다. 핵심은 "읽기 가능한 고밀도"이며, 행 간격과 타이포 대비로 리듬을 만든다.

## 7. 타이포그래피 취향 분석

### 7.1 강한 위계
헤드라인/섹션 타이틀/행 텍스트/보조 메타의 단계가 명확해야 한다. 크기 차이뿐 아니라 굵기, 자간, 대소문자 규칙, 숫자 정렬(탭형 숫자) 등으로 계층을 강화한다.

### 7.2 정보 신뢰를 만드는 활자 태도
법률·행정 성격의 데이터에서는 "장식적 개성"보다 "정확해 보이는 문장 운용"이 중요하다. 다만 관료적 포털처럼 무표정하지 않게, 제목부에서 편집적 캐릭터를 부여한다.

### 7.3 한국어/영문 혼용 최적화
Paradiso 맥락상 한국어 중심이지만, 비자 코드/영문 고유명사/기관명이 함께 등장한다. 따라서 한글 본문 가독성과 라틴 문자 숫자 정렬의 균형이 핵심이다.

## 8. 색상 및 표면감 분석

### 8.1 웜 뉴트럴 기반
기본 표면은 백색 일변도보다 약간 따뜻한 오프화이트·웜그레이 계열이 적합하다. 이는 차가운 데이터 인상을 완화하되, 귀엽거나 라이프스타일 브랜드처럼 보이지 않게 한다.

### 8.2 저채도 대비 + 기능적 포인트
포인트 컬러는 다채색 확장보다 소수의 기능적 강조색(링크, 상태, 경고)에 집중한다. 메인 정체성은 채도가 아니라 타이포와 그리드에서 확보한다.

### 8.3 얇은 경계선의 적극 활용
카드 그림자 대신 얇은 디바이더 라인으로 위계와 구역을 정의한다. 이는 "index-like rows" 감성을 강화한다.

## 9. 인터랙션/모션 취향 분석

### 9.1 절제된 반응성
호버, 포커스, 확장 동작은 존재하되 과시적 모션은 지양한다. 빠른 응답, 짧은 거리, 명확한 상태 변화가 중요하다.

### 9.2 텍스트 중심 인터랙션
버튼 애니메이션보다 행 하이라이트, 링크 언더라인 전환, 정렬/필터 상태 표기처럼 **읽기 흐름을 돕는 마이크로 인터랙션**이 우선이다.

### 9.3 탐색 우선 모션
모션은 브랜딩 쇼케이스가 아니라 탐색 비용 감소에 기여해야 한다. 예: 섹션 점프 시 위치 인지, 필터 적용 시 결과 영역의 안정적 갱신.

## 11. Paradiso에 적용할 원칙

1. **타이포가 레이아웃을 이끈다**: 컴포넌트 모양보다 텍스트 위계 설계가 선행되어야 한다.
2. **카드 대신 인덱스 행 구조를 기본값으로 둔다**.
3. **얇은 디바이더와 정교한 여백으로 구획한다**.
4. **웹사이트 우선 편집 체계**(특히 데스크톱)로 화면을 설계한다.
5. **웜 뉴트럴 바탕 + 저채도 포인트**로 표면 온도를 조정한다.
6. **모션은 기능 중심으로 절제**한다.
7. **시빅-테크 신뢰감 + 스튜디오 편집감**을 동시에 유지한다.

## 13. 새 디자인 컨셉: Editorial Civic Index

### 13.1 컨셉 정의
**Editorial Civic Index**는 "행정 정보를 읽기 좋게 편집한 공공성 있는 웹 인덱스"를 의미한다. 핵심은 '검색 도구'가 아니라 '신뢰 가능한 읽기·탐색 환경'이다.

Paradiso는 다음처럼 느껴져야 한다: **타이포그래피 주도형 에디토리얼 시빅-테크 웹 인덱스, 한국 비자·체류 정보를 위한.**

### 13.2 Paradiso는 무엇이 아닌가

다음 중 하나처럼 보이는 결과물은 실패다.

| 잘못된 방향 | 이유 |
|-------------|------|
| 아카이브(보관소) | 전시·보존 톤은 탐색 도구와 맞지 않는다. 사용자는 정보를 '보관'하러 오는 게 아니라 '찾으러' 온다. |
| AI 챗봇 | 대화 버블, 타이핑 표시, 채팅 인터페이스가 전면에 나오면 정보 신뢰가 무너진다. |
| SaaS 랜딩 페이지 | 가입 유도, 특장점 카드, 가격 플랜, CTA 버튼이 전면에 나오면 행정 정보 도구가 아니다. |
| 모바일 앱 | 큰 둥근 카드, 바텀 탭, FAB, 챗 버블은 웹 인덱스가 아니다. |
| 정부 포털 | 무표정한 테이블, 레거시 폰트, 공문서 레이아웃은 신뢰가 아니라 두려움을 준다. |
| 법률 데이터베이스 | 과밀 테이블, 법조문 그대로 나열, 코드·조문 번호만 강조하는 UI는 진입 장벽이다. |
| 카드 대시보드 | 정보를 모두 카드로 만들면 스캔 속도가 떨어지고 모바일 앱처럼 보인다. |

### 13.3 "Soft Source Archive" 명시적 폐기

**"Soft Source Archive"는 본 프로젝트의 디자인 컨셉으로 더 이상 사용하지 않는다.**

폐기 이유:
- Figma Make가 이 문구를 반복적으로 박물관, 기록보관소, 라이브러리 UI로 해석하는 오류를 일으켰다.
- "Soft"라는 형용사가 의도한 "따뜻한 표면감"을 전달하는 데 실패하고, 대신 모호한 감성·아카이브 분위기를 불러온다.
- "Archive"는 탐색이 아니라 보존을 암시하여 Paradiso의 목적(빠른 행정 정보 탐색)과 충돌한다.

모든 Figma Make 프롬프트에서 "Soft Source Archive", "archive", "museum", "library", "보관소", "아카이브"라는 단어를 사용하지 않는다. 대신 **"Editorial Civic Index"**를 사용한다.

### 13.4 시각 원칙
- 강한 제목부와 명확한 서브헤드
- 목록 중심 본문(행/열/인덱스)
- 얇은 선 기반 구분
- 절제된 강조색
- 문서형 리듬을 가진 섹션 전개

### 13.5 경험 원칙
- 사용자는 빠르게 스캔하고, 필요한 맥락을 즉시 확인할 수 있어야 한다.
- 정보 단위는 "읽히는 순서"에 맞춰 편집되어야 한다.
- UI의 존재감보다 정보 이해 속도를 우선한다.

## 14. Figma Make용 최종 지시문

다음은 Figma Make 프롬프트 입력창에 직접 붙여넣을 수 있는 최종 지시문이다.

---

**[Figma Make 지시문 — 복사·붙여넣기용]**

> Paradiso를 한국 비자·체류 정보를 위한 **typography-led editorial civic-tech web index**로 리디자인하라.
>
> **컨셉 이름: Editorial Civic Index.** "Soft Source Archive", "archive", "museum", "library" 같은 단어는 이 디자인과 관계없다. 절대 사용하지 말 것.
>
> **데스크톱 우선으로 설계하라.** 모바일은 데스크톱 레이아웃의 반응형 축소판이다. 모바일 앱을 기준으로 설계하지 말 것.
>
> **아래 9개 프레임을 반드시 생성하라:**
> 1. Desktop — 랜딩 상태 (히어로: 타이포그래피 주도형 인덱스, 검색 토글, 행정 도구 인덱스 행, 입국 전/후 경로 항목, 추천 키워드 행, 출처 면책 고지)
> 2. Desktop — 검색 상태 (스티키 헤더 + 비자 레코드 인덱스 행 목록 + 쿼리 필터 행)
> 3. Desktop — 소스 레코드 상세 (비자 종류 상세: 조건, 서류, 기관 참조, 인라인 확장)
> 4. Desktop — AI 안내 보조 레이어 또는 슬라이드인 패널 (검색 결과 내 비노출 진입, 모달 또는 패널 형태)
> 5. Desktop — 행정 도구 모달 또는 패널 (관할 관서 조회, 체류기간 계산 등)
> 6. Mobile — 랜딩 상태 (데스크톱 레이아웃의 단일 컬럼 축소판)
> 7. Mobile — 검색 상태 (인덱스 행 목록, 스티키 헤더)
> 8. 디자인 토큰 패널 (색상, 타이포그래피, 간격, 선 두께 토큰)
> 9. 컴포넌트 시트 (인덱스 행, 디바이더, 칩, 배지, 헤더, 면책 고지 컴포넌트)
>
> **레이아웃 규칙:**
> - 소스 레코드 목록은 카드 그리드가 아니라 **인덱스 행(row)** 구조다. 각 행은 좌(비자 코드), 중(명칭·요약), 우(카테고리·상태 배지)로 구성.
> - `border-radius`는 행 컴포넌트에서 0–4px로 최소화한다. 큰 둥근 카드 금지.
> - 카드 그림자 대신 **1px 수평 디바이더 라인**으로 행을 구분한다.
> - 랜딩 화면은 검색창 하나만 있는 단순 검색 앱이 아니다. 검색 토글 버튼, 행정 도구 인덱스, 입국 전/후 경로 항목, 추천 키워드 행, 출처 면책 고지가 모두 보여야 한다.
>
> **타이포그래피 규칙:**
> - 레이아웃을 이끄는 것은 이미지나 일러스트가 아니라 타이포그래피 위계다.
> - 헤드라인 / 섹션 타이틀 / 행 텍스트 / 메타 정보의 4단계 위계를 명확히 구현한다.
> - 한글 본문 가독성과 비자 코드(라틴 문자) 숫자 정렬(tabular nums)을 동시에 고려한다.
>
> **색상 규칙:**
> - 기본 표면은 순백색이 아닌 **웜 뉴트럴 오프화이트** (예: #F8F6F1 또는 유사 계열).
> - 포인트 컬러는 1–2개의 기능적 강조색(링크, 상태)만 허용한다. 다채색 카드 배경 금지.
> - 어두운 테마는 웜 다크(#0f1a17 또는 유사 계열)를 기본으로 한다.
>
> **AI 레이어 규칙:**
> - AI는 주요 진입점이 아니라 **보조 레이어**다.
> - 고정 FAB(Floating Action Button) 형태로 화면에 항상 노출하지 말 것.
> - AI 진입점은 검색 결과 하단 텍스트 링크 또는 레코드 상세 내 비노출 액션으로 처리한다.
> - `ai.html`은 별도 모달 또는 슬라이드인 패널로만 등장한다.
>
> **금지 사항:**
> - 박물관/아카이브/라이브러리 메타포 사용 금지
> - 제너릭 SaaS 랜딩 페이지 스타일 금지
> - 챗봇 UI(말풍선 버블, 타이핑 점 표시) 전면 노출 금지
> - 정부 포털 클론 스타일 금지
> - 모바일 앱 패턴(큰 둥근 카드, 바텀 탭, 전체 화면 FAB) 금지
> - 코드 생성 금지 — 비주얼 프레임만 생성할 것

## 15. 구현 전 체크리스트

아래 항목은 합격(PASS)/불합격(FAIL) 기준으로 평가한다. 모든 항목이 PASS여야 리뷰를 진행한다.

### 15.1 구조 및 제품 레이어 완전성

- [ ] **PASS/FAIL** — 데스크톱 랜딩 프레임이 단순 검색창 하나 이상을 보여주는가? (검색 토글 버튼, 행정 도구 목록, 입국 전/후 경로 항목, 추천 키워드, 출처 면책 고지가 모두 보여야 PASS)
- [ ] **PASS/FAIL** — 검색 상태(searched state) 프레임이 별도로 존재하는가? (스티키 헤더 + 인덱스 행 결과 목록이 보여야 PASS)
- [ ] **PASS/FAIL** — 소스 레코드 상세 프레임이 존재하는가?
- [ ] **PASS/FAIL** — AI 안내 레이어가 보조 모달 또는 슬라이드인 패널로만 등장하는가? (전면 노출 챗봇이면 FAIL)
- [ ] **PASS/FAIL** — 행정 도구 모달 또는 패널 프레임이 존재하는가?
- [ ] **PASS/FAIL** — 모바일 랜딩 및 모바일 검색 상태 프레임이 각각 존재하는가?
- [ ] **PASS/FAIL** — 디자인 토큰 패널과 컴포넌트 시트가 존재하는가?

### 15.2 레이아웃 원칙 준수

- [ ] **PASS/FAIL** — 소스 레코드 목록이 카드 그리드보다 인덱스 행 구조에 가까운가? (큰 둥근 카드 그리드이면 FAIL)
- [ ] **PASS/FAIL** — 행 간 구분이 카드 그림자보다 얇은 디바이더 라인으로 처리되었는가?
- [ ] **PASS/FAIL** — 데스크톱 레이아웃 완성도가 모바일보다 높은가? (데스크톱이 더 풍부한 정보 구조를 가져야 PASS)
- [ ] **PASS/FAIL** — 행정 도구 목록이 2열 카드 그리드보다 인덱스 행 형태인가?

### 15.3 타이포그래피 원칙 준수

- [ ] **PASS/FAIL** — 타이포그래피 위계(헤드라인/섹션/행 텍스트/메타)가 즉시 구분되는가?
- [ ] **PASS/FAIL** — 레이아웃의 시각적 중심이 이미지/일러스트가 아닌 타이포그래피인가?
- [ ] **PASS/FAIL** — 비자 코드 등 라틴 숫자에 탭형 숫자(tabular nums)가 적용되었는가?

### 15.4 컨셉 정합성

- [ ] **PASS/FAIL** — 박물관/아카이브/라이브러리 느낌이 없는가? (있으면 FAIL)
- [ ] **PASS/FAIL** — 챗봇/AI 어시스턴트가 전면에 노출되지 않는가?
- [ ] **PASS/FAIL** — 제너릭 SaaS 랜딩 페이지처럼 보이지 않는가?
- [ ] **PASS/FAIL** — 정부 포털 클론처럼 보이지 않는가?
- [ ] **PASS/FAIL** — 모바일 앱(큰 둥근 카드, 바텀 탭, 전면 FAB) 패턴이 아닌가?
- [ ] **PASS/FAIL** — 법률 데이터베이스형 과밀 테이블 UX가 아닌가?

### 15.5 법적 정합성

- [ ] **PASS/FAIL** — 디자인이 비자 승인이나 행정 결정을 보장하는 것처럼 보이지 않는가?
- [ ] **PASS/FAIL** — 출처 면책 고지가 경고 박스가 아닌 편집 주석 형태로 표현되는가?
- [ ] **PASS/FAIL** — AI 진입점이 "법적 판단 제공"이 아닌 "참고 안내" 톤으로 표현되는가?
